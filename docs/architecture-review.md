### 4. Search check

- **[Layer 1]** Cloudflare Durable Objects + WebSocket：官方推荐 Hibernation API（节省 GC 成本），用 `state.acceptWebSocket(ws)` 而不是自己维护 `ws` 数组。**架构影响**：DO 类里不要写 `this.sockets = new Set()`，用 `state.getWebSockets()` 查询。
- **[Layer 1]** D1 + Prepared Statements：所有 SQL 必须用 `prepare()` + bind，禁止字符串拼接（SQL injection 防护）。
- **[Layer 1]** Stripe webhook 幂等：用 event id 做 idempotency key，存到 D1 `stripe_events` 表防止重复处理。
- **[Layer 2]** D1 写入延迟：D1 是 SQLite-based，写延迟 ~10-50ms，跨 region 读有 replication lag。**架构影响**：消息写入后立即读要走 DO buffer（强一致），不走 D1。D1 只做历史回放和搜索。
- **[Layer 2]** Hono on Workers 的 WebSocket：Hono 4.x 原生支持 `upgrade`，不需要额外库。
- **[EUREKA 候选]**：用 DO 作为"热消息缓存 + 实时广播"，D1 只做冷存储和搜索 —— 这其实和 Cloudflare 官方"chat app"示例一致，但加了一个反直觉点：**消息先写 D1，再让 DO 异步广播**（不是"DO 持消息 → 异步落 D1"）。理由：D1 写失败比 DO 状态丢失更容易恢复（DO 单实例 crash 会丢未持久化的 buffer），所以持久化优先于实时广播。30ms 的写延迟对用户感知可接受。

#### Issue 1: DO 单实例是 channel 的单点故障 + 扩展上限

**Finding** `[P1] (confidence: 9/10)` — DO 是 per-channel 单实例，crash 后状态丢失（除非用 storage API 持久化），且单 DO 有 CPU/wall time quota（Workers 限制）。

**代行 AskUserQuestion**：
- A) 现在就做 DO 分片（多实例 fail-over）—— effort (human: ~2 周 / CC: ~2 天)，risk High，维护 burden 高（需要跨实例状态同步）
- B) 用 DO Hibernation API + storage.put() 持久化关键状态到 DO storage —— effort (human: ~3 天 / CC: ~4 小时)，risk Low，维护 burden 低
- C) 什么都不做，监控告警，出问题再处理 —— effort 0，risk Med（用户感知：单 channel 在 DO 重启时掉线 5-30 秒）

**Recommendation**: Choose B. 用户的 user CLAUDE.md §3 "不过度抽象、不提前设计灵活性" + skill "incremental over revolutionary"。B 是最小可逆投入，A 是 over-engineering。

**代行决策**：选 B。写到 NOT in scope: A（DO 分片，等 1000 channel/天 时再做）。

#### Issue 2: D1 跨 region 读延迟 + 写入瓶颈

**Finding** `[P2] (confidence: 7/10)` — D1 primary region 写入，其他 region 异步复制。读消息历史时，跨 region 用户可能看到 ~100-500ms 延迟。

**代行 AskUserQuestion**：
- A) D1 read replica（Cloudflare roadmap 功能，部分 region 可用）—— effort (human: ~1 周 / CC: ~1 天)，risk Med（功能还在演进）
- B) DO 持最近 N 条消息作为热缓存，D1 只做历史搜索 —— effort (human: ~3 天 / CC: ~4 小时)，risk Low，已在 EUREKA 候选里覆盖
- C) 不优化，监控 P95 —— effort 0，risk Low（当前用户量 D1 延迟不是瓶颈）

**Recommendation**: Choose B (already covered by EUREKA)。理由：和 Issue 1 的 B 同一个 DO buffer 设计一并解决，零额外成本。

**代行决策**：选 B。

#### Issue 3: Stripe webhook 幂等性

**Finding** `[P1] (confidence: 10/10)` — Stripe 会重试 webhook（最多 3 天），如果 Workers 处理慢或抛错，同一 event 会被收到多次。不幂等 = 同一订阅被开通多次 / 邮件发多次。

**代行 AskUserQuestion**：
- A) 写 `stripe_events` 表存已处理 event id，处理前检查 —— effort (human: ~1 天 / CC: ~30 分钟)，risk Low，是 [Layer 1] 标准做法
- B) 用 Workers KV 做幂等 key（TTL 7 天）—— effort (human: ~3 天 / CC: ~1 小时)，risk Low，但 KV 有最终一致性窗口（罕见但存在）
- C) 信任 Stripe-Signature 头 + 不做幂等 —— effort 0，risk High（数据完整性）

**Recommendation**: Choose A. D1 SQL 是强一致，event id 唯一索引。符合 user CLAUDE.md §7 "不注释报错绕过问题，定位根本原因"。

**代行决策**：选 A。

#### Issue 4: WebSocket 鉴权 + channel 访问授权

**Finding** `[P1] (confidence: 9/10)` — WS 连接不能用普通 HTTP Authorization header（浏览器 API 限制）。常见反模式：把 token 放 query string（会被日志/CDN 泄漏）。

**代行 AskUserQuestion**：
- A) 第一次 WS 握手用 query string 传 short-lived ticket（5 秒有效，单次使用，用后作废），后续消息用 DO 内的 session 校验 —— effort (human: ~3 天 / CC: ~6 小时)，risk Low，[Layer 1] 推荐模式（Slack/Discord 都这么干）
- B) 用 cookie（同源 + SameSite=Strict）—— effort (human: ~1 天 / CC: ~2 小时)，risk Low，但跨子域 / 第三方 cookie 限制在 Safari 上踩坑
- C) 不鉴权 WS，依赖 Worker 路由前置鉴权 —— effort 0，risk Critical（任何人扫到 channel id 可加入）

**Recommendation**: Choose A. 短期 ticket 模式是 [Layer 1]，且对 Cloudflare 边缘架构最友好（cookie 在跨 region 边缘节点一致性差）。

**代行决策**：选 A。

#### Issue 5: 消息持久化 vs 实时广播的顺序

**Finding** `[P1] (confidence: 8/10)` — design 没写清楚"先持久化再广播"还是"先广播再持久化"。前者：用户感知慢 30ms 但数据不丢；后者：用户感知快但 DO crash 时丢消息。

**代行 AskUserQuestion**：
- A) 先写 D1，成功后再让 DO 广播 —— effort (human: ~1 天 / CC: ~1 小时)，risk Low，已在 EUREKA 候选
- B) 先广播，异步写 D1 —— effort (human: ~2 天 / CC: ~2 小时)，risk High（消息丢失）
- C) 只写 DO buffer，定时批量 flush 到 D1 —— effort (human: ~4 天 / CC: ~4 小时)，risk Med（DO crash 丢未 flush 的批量）

**Recommendation**: Choose A. 与 EUREKA 候选一致，且符合 user CLAUDE.md "不出安全事故" 的红线（数据丢失对付费用户是安全事故）。

**代行决策**：选 A。

#### Issue 6: D1 schema 设计 - 消息表索引

**Finding** `[P2] (confidence: 8/10)` — 消息表查询路径是 `WHERE channel_id = ? ORDER BY created_at DESC LIMIT 50`。如果没有 `(channel_id, created_at)` 复合索引，单 channel 超过 10 万条消息后查询会扫表。

**代行 AskUserQuestion**：
- A) 加复合索引 `(channel_id, created_at DESC)` —— effort (human: ~30 分钟 / CC: ~5 分钟)，risk Low，[Layer 1]
- B) 加单列索引 `channel_id` + `created_at` 各一个 —— effort 同 A，risk Med（SQLite 优化器可能不选复合路径）
- C) 不加，等查询慢了再加 —— effort 0，risk Low 短期 / High 长期

**Recommendation**: Choose A. 索引成本几乎 0，符合 Boil the Lake 原则。

**代行决策**：选 A。

**Architecture Review 完成，6 个 issue，代行全部接受推荐方案。**

### 4. Performance Review

#### Issue 10: 消息列表 N+1 查询

**Finding** `[P2] (confidence: 8/10)` — `GET /messages/:channelId` 实现里，如果消息列表渲染时要 join 用户信息（avatar、name），naive 实现是每条消息查一次 user，50 条消息 = 51 次 D1 查询。

**代行 AskUserQuestion**：
- A) D1 JOIN 一次查询：`SELECT m.*, u.name, u.avatar FROM messages m JOIN users u ON m.user_id = u.id` —— effort (human: ~30 分钟 / CC: ~5 分钟)，risk Low
- B) 应用层 batch：先查 50 条消息，再 `WHERE user_id IN (...)` 批量查 users —— effort 同 A，risk Low，但代码多
- C) 不优化，等慢了再说 —— effort 0，risk Med

**Recommendation**: Choose A. SQLite JOIN 是 [Layer 1]，且 D1 单次 round-trip 比 2 次快。

**代行决策**：选 A。

#### Issue 11: WebSocket 心跳 / 内存泄漏

**Finding** `[P3] (confidence: 7/10)` — 长连接 WebSocket 如果不做 ping/pong，proxy/CDN 会在 60-90s idle 后断开。前端如果没清理 reconnect 定时器，会内存泄漏。

**代行 AskUserQuestion**：
- A) 服务端 30s ping，前端 60s 没收到 pong 则重连 —— effort (human: ~1 天 / CC: ~2 小时)，risk Low
- B) 客户端定时发 heartbeat 消息 —— effort 同 A，risk Low
- C) 不做，依赖 Cloudflare 默认 —— effort 0，risk Med（Cloudflare WS idle timeout 100s，刚好踩边界）

**Recommendation**: Choose A。30s ping 是行业 [Layer 1]。

**代行决策**：选 A。

#### Issue 12: Bundle size — Workers 1MB 限制

**Finding** `[P3] (confidence: 8/10)` — Cloudflare Workers 单 bundle 压缩后 ≤ 1MB。Hono + Zod + Stripe SDK + better-auth 加起来如果不 tree-shake，可能超限。

**代行 AskUserQuestion**：
- A) CI 加 `wrangler deploy --dry-run` + bundle size 检查 —— effort (human: ~1 小时 / CC: ~15 分钟)，risk Low
- B) 拆成两个 Worker（auth Worker + chat Worker）—— effort (human: ~1 周 / CC: ~1 天)，risk Med（增加运维复杂度）
- C) 不检查 —— effort 0，risk Med（部署时才发现就晚了）

**Recommendation**: Choose A。CI gate 成本极低，符合 Boil the Lake。

**代行决策**：选 A。

**Performance Review 完成，3 个 issue，代行全部接受推荐方案。**

| S11: Shared types (Zod schemas) | `packages/shared/` | — |
| S12: E2E playwright suite | `e2e/` | S7, S8, S9 |

#### Parallel Lanes

- **Lane A (backend core)**: S1 → S2 → S3 → S4 → S5 → S6（共享 `src/server/`，必须顺序）
- **Lane B (shared types)**: S11 → S10（独立，与 A 并行启动）
- **Lane C (frontend)**: 等 A 完成 S2/S3/S4 的 API contract 后启动 → S7 → S8 → S9（共享 `src/web/`，顺序）
- **Lane D (E2E)**: 等 C 完成 → S12

#### Execution Order

1. **并行启动**：Lane A + Lane B（两个独立 worktree）
2. **合并 B 到 A**：S11 完成后，A 用 B 的类型（cherry-pick 或合并）
3. **A 完成 S2/S3/S4 后**：启动 Lane C（第三个 worktree），C 依赖 A 已稳定的 API contract
4. **A 完成 S5 后**：C 可以做 S9
5. **C 完成后**：启动 Lane D（E2E），D 依赖 C 的 UI

#### Conflict Flags

- Lane A 和 Lane B 都会动 `packages/shared/`（A 引用，B 定义）—— **冲突风险低**，B 先定义好再 merge 给 A。
- Lane C 依赖 A 的 API contract，A 中途改 contract 会破坏 C —— **缓解**：A 的 API contract 用 S11 的 Zod schema 锁定，schema 一旦 publish 给 C 就 freeze，后续改动走 deprecation。
