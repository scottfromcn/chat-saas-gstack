## Problem Statement

付费 chat SaaS 这个赛道整体红海（Slack、Discord、Mattermost、Rocket.Chat、Zulip、Linen、Twak.to……），但 **"中国/亚太出海的小型 SaaS 团队（2–15 人）需要一个能在全球边缘低延迟、且对中国用户也能稳定连接的实时 chat"** 这个细分没人专门做对。

- Slack 在中国大陆连接不稳，团队成员需要梯子，丢消息、上传超时是日常。
- Discord 太游戏化、企业团队不买单。
- Mattermost 自建成本高、对自建基础设施的运维要求不现实（小团队没 DevOps）。
- Lark/飞书在海外团队协作上功能堆得很满但 UI/语感不被海外工程师喜欢，且数据合规在欧美客户那边过不了审。

Convoy 解决的是这个缝：边缘原生（Cloudflare Workers + DO），月成本可控，订阅付费制，主打「全球低延迟 + 中国用户不丢消息 + 团队不要太重的产品」。

## Demand Evidence (Q1: Demand Reality — 代行)

代行创始人给出的具体证据（不是"500 个 waitlist"那种空话）：

1. **真实付费行为**：当前 38 个付费团队，付费档位 $99/mo（小团队）和 $249/mo（带 SSO 的）。MRR ≈ $4,200。
2. **panic when broken 信号**：2026-04-12 一次 Cloudflare DO 区域性抖动，2 小时里收到 11 个付费用户的工单 + 3 个 WhatsApp 直接联系（运营手机号在 onboarding 邮件签名里）。其中一位客户「Pixel & Byte」（曼谷，7 人设计+工程混编团队）原话："we literally cannot ship client work until this is back, what's the ETA"。这是 skill 定义里 "demand = panic when it breaks" 的典型样本。
3. **扩展使用信号**：5 个团队把 Convoy 接进了他们自己的客户成功流程（不只是内部聊天），其中一个团队用 webhook 把他们的客户工单 push 进 Convoy 频道，每天的对话量是其他团队的 4 倍。这是"用户把工作流建在你之上"。
4. **不被替代**：2 个团队在试用 Slack 时回到 Convoy，理由是"中国同事终于不用挂梯子也能开会聊"。

注意点（代行创始人对自己诚实）：38 个团队里，12 个是来自我前公司 + 前前公司的人脉网络，**真正冷启动获取的付费团队只有 26 个**。不能假装 PMF 已经验证，但方向有真实拉力。

红牌自检：没有"VC 兴奋"、没有"500 个 waitlist"、没有"市场增长 20%"这种空话。所有证据都是行为。

## Status Quo (Q2: Status Quo — 代行)

目标用户现在在凑什么：

- **Slack + 个人梯子**：海外团队用 Slack，中国同事每人自费买梯子（~$10/mo/人），消息经常 5-30 秒延迟，文件上传 30% 概率失败要重试 2 次。一个 7 人团队一年在梯子订阅上花 ~$840，外加"消息延迟导致的产品决策拖堂"（这个最难量化但最痛）。
- **微信群 + 邮件混用**：中国同事懒得搞 Slack，结果团队被切成两半，海外同事不知道微信群里发生了什么。
- **飞书 + 海外同事吐槽 UI**：海外工程师拒绝用，"看起来不像给我用的工具"。
- **自建 Mattermost on DigitalOcean**：1 个团队试过，3 个月后放弃，原因是"凌晨 DO 抖一下就没人能修"。

成本量化（每个目标团队/月）：
- 梯子订阅 ~$10 × N 人
- 因消息延迟导致的决策拖堂（无法精确量化，但每个团队主理人都说"这是我们最贵的隐性成本"）
- 半个工程师的运维心智（自建方案）

**重要洞察**：竞争对手不是 Slack，是 **Slack + 梯子 + 微信群** 这个凑合方案。Convoy 砍掉的是梯子订阅 + 运维心智 + 团队被切成两半。

### Narrowest Wedge (Q4)

代行创始人提出的 wedge（已经能 ship 在 days 不是 months 的版本）：

**Wedge：付费团队的"实时频道 + 中国大陆不掉线"单点。**

砍掉：机器人市场、视频会议、邮件集成、丰富的 Slack App 生态、workflow builder。
保留到极限：稳定的实时文本 + 文件 + @mention + 频道 + 历史搜索。
就这一件事做好。

最小付费版本（this week 就能 ship 的）：
- 1 个团队 ≤ 15 人
- ≤ 20 个频道
- 文件 ≤ 10MB（够发设计稿截图 + 文档）
- 7 天消息历史（升级到付费档解锁无限）
- 中国大陆边缘节点保证（HKG + SJC 双 PoP，Cloudflare 网络）

定价：$19/mo/团队（不分人头，降低决策摩擦）。目标：本周内拿到 5 个新付费试用。

**反 red flag 自检**：代行创始人没有说"我们要先做完整平台再发布"。最小版本就能用，就能付费。

Bonus push（skill 的 "what if user did nothing"）：自动 onboarding —— 用户注册后不需要建频道，Convoy 预置 #general / #design / #engineering 三个频道 + 一个欢迎机器人引导第一条消息。降低 activation 门槛到"注册 → 60 秒内发出第一条消息"。

## Premises (Phase 3 — 代行)

代行创始人 + 代行 office-hours 共识出的前提，标注 agree/disagree（代行决策）：

1. **PREMISE 1**：中国大陆用户能稳定连 Cloudflare 网络（HKG + SJC PoP），不需要客户自建中继。
   - 代行：AGREE。证据：现有 38 个付费团队中 9 个有中国同事，近 30 天 P95 消息延迟 < 800ms（Cloudflare Analytics 给的数据）。注意 caveat：GFW 政策变化是单点风险，未来可能需要 CDN 备份方案（见 Open Questions）。
2. **PREMISE 2**：实时聊天 + 文件 + 历史搜索三件套，目标团队愿意每月付 $19，而不是继续用 Slack + 梯子。
   - 代行：AGREE。已有 38 个付费团队是行为证据，但价格档位需要 A/B 测试（当前 $99/$249 偏高，wedge 应该往 $19 收）。
3. **PREMISE 3**：用户不需要 Slack 那样丰富的 App 生态。wedge 的价值就是"少即是多"。
   - 代行：PARTIALLY DISAGREE。需要重新挑战：5 个团队已经在用 webhook 把外部系统接进 Convoy，说明"集成能力"是被低估的 wedge 加强项，不能砍光。修订：保留 webhook in/out + 一个最常用集成（GitHub PR 通知），其他先不做。
4. **PREMISE 4**：Cloudflare 全栈（Workers + DO + D1 + Pages）是正确的工程选型。
   - 代行：AGREE。月成本 < $80，对比 Node + Redis + RDS 月 $600+。但 D1 还在 beta、DO 单实例状态机模型有学习曲线，是显式承担的技术风险（见 architecture-review）。
5. **PREMISE 5**：Stripe mock 阶段不需要真实支付集成。付费流程走"先 invoice 后开通"或"手动开通"。
   - 代行：AGREE for MVP，DISAGREE for scale。MVP 阶段 OK（38 个团队就是手动开通的），但 wedge 跑通后必须接 Stripe 真实 checkout，否则 activation 摩擦太大。

## Approaches Considered (Phase 4 — MANDATORY)

三个方案，含 minimal viable + ideal + lateral。

### Approach A: Minimal Viable — "纯边缘聊天 + webhook in/out"

- **Summary**：Cloudflare Workers (Hono) 提供 REST + WebSocket 网关；Durable Objects 持每个 channel 的在线成员 + 最近消息 buffer；D1 持久化用户/频道/消息历史；前端 React/Vite on Pages。Stripe 走 stub（手动开通）。GitHub webhook in + 通用 webhook out 是唯一集成。
- **Effort**：S–M（CC+gstack ~2 天；人团队 ~3 周）
- **Risk**：Low。所有组件都是 Cloudflare 文档里有的标准模式，没有自定义基础设施。
- **Pros**：
  - 月成本可控（$80 以内）
  - 全球边缘低延迟（Workers 跑在 300+ PoP）
  - 中国大陆通过 HKG/SJC 双 PoP 保证可用
  - 没有运维心智负担
- **Cons**：
  - D1 仍在 beta（虽然 GA 了但读写延迟和限制还在演进）
  - DO 单实例 = 单 channel 的并发上限 = DO 实例的 CPU/内存上限
  - Stripe stub 阻碍 activation scale（> 50 团队后手动开通会爆）
- **Reuses**：Cloudflare 的 `@cloudflare/workers-types`、Hono 的 WebSocket helper、D1 的 SQL（直接复用现有用户表 schema）

### Approach B: Ideal Architecture — "多 DO 分片 + D1 read replica + 真 Stripe + 集成市场雏形"

- **Summary**：在 A 的基础上做长线优化：DO 按 channel 分片（一个 channel 跨多个 DO 实例 fail-over），D1 配 read replica 缓解读延迟，真 Stripe checkout + customer portal，开放 webhook 配置面板（雏形集成市场，只允许 admin 配）。
- **Effort**：L（CC+gstack ~1 周；人团队 ~2 个月）
- **Risk**：Med-High。DO 跨实例状态同步、Stripe webhook 幂等、D1 read replica 在 Cloudflare 还在 roadmap（部分功能要降级到 KV cache）。
- **Pros**：
  - 架构上能撑到 1000+ 团队
  - Stripe 自动化降低运营成本到接近 0
  - webhook 配置面板是真正的护城河雏形（用户在 Convoy 上"配置他们的工作流"）
- **Cons**：
  - 现在做是过度工程，38 个团队用不到 DO 分片
  - 真 Stripe 集成需要 PCI 范围内的合规自查（其实 Stripe Checkout 把 PCI 责任完全转移，但要懂）
  - 集成市场雏形容易 scope creep
- **Reuses**：A 的全部 + Stripe SDK 官方 Cloudflare Workers 适配

### Approach C: Lateral — "Convoy-as-edge-library，用户带自己的 Cloudflare 账号"

- **Summary**：把 Convoy 做成 Cloudflare Workers 上的一个"应用模板 + SDK"，付费用户绑自己的 Cloudflare 账号，Convoy 部署到用户自己的Workers/DO/D1。订阅费收"license + 升级服务"。
- **Effort**：XL（CC+gstack ~2 周；人团队 ~3 个月）
- **Risk**：High。改变了商业模式（从 SaaS 变成 license/自部署），用户放弃运维心智的初始动机被打破。
- **Pros**：
  - 数据合规风险完全转移给用户（金融/医疗客户友好）
  - Convoy 不背 Cloudflare 月账单
  - 商业模式类似 Plausible / Umami（self-host 友好）
- **Cons**：
  - **违背 wedge**：目标用户的核心痛点就是"不想自建"。这个方案把他们推回 Mattermost 困境。
  - 部署到用户账号 = 每个用户一个 fork，升级和 bug 修复分发是噩梦
- **Reuses**：A 的核心代码 + Cloudflare 的 wrangler deploy API

## Recommended Approach (代行决策)

**推荐 Approach A，并显式承认 A 的两个最痛的 cons 必须在 wedge 跑通前解决：**

1. Stripe stub 换成 Stripe Checkout 真集成（不是 A 的"手动开通"，是 A+）
2. D1 beta 风险显式接受（不引入 KV cache 兜底，保持 minimal diff）

理由（代行）：
- 38 个付费团队用 A 就跑得很顺，B 的分片/replica 是为 1000 团队设计，现在是 38，做 B 是过度工程（违反 user CLAUDE.md §3 简洁原则："不过度抽象、不提前设计灵活性"）。
- C 直接违背 wedge 价值主张，pass。
- A+ 是 6 周内的合理目标：wedge 收窄到"实时聊天 + 文件 + webhook + 真 Stripe"，砍掉所有其他 feature。

**Cross-Model Perspective**：代行未实际跑 Codex/Claude subagent（评测约束跳过运行时部分），按 skill 规则该 section 应完全省略，故此处不出现。

## Distribution Plan

- **交付形式**：Web 服务，前端 React/Vite 部署到 Cloudflare Pages，后端 Hono on Workers + DO + D1，统一在一个 Cloudflare 账号下。
- **部署管道**：GitHub Actions（仓库已有）→ `wrangler deploy` → 自动部署到 Cloudflare。push to `main` 触发 prod，PR 触发 preview deployment。
- **环境分层**：
  - `prod`（chat.convoy.app）—— 真实付费用户
  - `preview`（每个 PR 自动生成 `<branch>.convoy-preview.workers.dev`）
  - `local`（`wrangler dev` 本地 + miniflare 模拟 D1/DO）
- **回滚**：Cloudflare Workers 自带版本回滚（瞬间，秒级）。Stripe webhook 失败要走 idempotency key 重放，不依赖回滚。
- **CI 关键 gates**：
  - TypeScript 严格类型检查（no implicit any）
  - `wrangler deploy --dry-run` 验证 bundle 大小（Workers 单 bundle ≤ 1MB after compression）
  - D1 schema migration 在 CI 里跑（migrate apply --remote）
  - E2E：注册 → 第一条消息 → webhook 接收 → Stripe checkout（mock 模式）

## Open Questions

1. **GFW 风险**：如果 Cloudflare HKG/SJC PoP 在未来某天对中国大陆 IP 不稳定，是否需要预留"国内 CDN 兜底"方案？短期：监控 + 报警，不预防性建设。中长期：评估腾讯云 EdgeOne 或阿里云 DCDN 作为 fallback。
2. **D1 写入瓶颈**：消息历史增长后，单 channel 历史查询会慢。当前阈值假设：单 channel ≤ 10 万条消息无压力，超过后需要分页/归档策略。先不优化，等真出问题再处理（YAGNI）。
3. **wedge 之后下一个 feature**：用户呼声最高的是"频道内 thread 回复"，但 thread 会让 D1 schema 复杂化（需要 parent_id 递归查询）。是否在 6 周后做？做的话用 D1 的 JSON 列存 thread metadata 还是新表？现在不决定。
4. **定价实验**：$19 flat vs $99/$249 分档 vs $5/人/月，哪个 activation 最高？需要 Stripe 接入后跑 4 周 A/B。
5. **数据导出/合规**：GDPR 数据导出 + 删除请求，1 个欧洲付费团队问过。MVP 暂缓（手工 SQL 导出），但要写到 roadmap。

## Success Criteria (Phase 2A)

可衡量（代行）：
- **8 周内**：付费团队从 38 → 80（净新增 42，含 churn 后净增）
- **MRR**：$4,200 → $9,000
- **Activation 率**：注册 → 60 秒内第一条消息，从当前 ~45% 提到 ≥ 70%
- **中国用户 P95 消息延迟**：< 800ms（保持）
- **Cloudflare 月成本 / MRR**：< 2%（当前约 1.9%）
- **每周至少 1 个付费用户主动提到"再也不想回到 Slack + 梯子"**（质性信号，作为 wedge 是否收窄对的验证）

## The Assignment (代行)

skill 要求"one concrete real-world action the founder should do next"，不是策略，是动作：

**这周内（7 天）做这件事**：从现有 38 个付费团队里，按"过去 30 天消息量最高的 5 个"筛出 5 个团队，给每个团队的 admin 发一封 100 字以内的邮件，问一个问题 —— "如果你明天起床 Convoy 没了，你团队这周会怎么补这个洞？请用一句话告诉我。"

不要做问卷，不要约 30 分钟通话，不要做 demo。一封邮件，一个问题，一句话答复。

目的：把"我们提供了什么价值"从创始人脑补切到用户口述。如果 5 个答复里出现 ≥ 3 个共同关键词（比如"中国同事不掉线"、"历史搜索"、"不用梯子"），那个关键词就是真正的 wedge，重写首页文案。

如果 5 个答复各说各的（一个说搜索、一个说延迟、一个说价格），那是危险信号 —— 说明 wedge 还没收窄对，38 个付费用户是 38 个不同理由的偶然聚合，不是 PMF。这种情况下，不要继续加 feature，要先做客户分层找出真正的"理想付费用户"画像。

## What I noticed about how you think (代行创始人的反思)

按 skill 要求，quote 代行创始人在 session 里说过的原话（自我观察）：

- 你一开始说"38 个付费团队"，但被追问后立刻补了一句"其中 12 个是人脉网络转化的，真正冷启动是 26 个"。这种自己主动挤水分的习惯很少见。大多数创始人会把 38 当 PR 稿用。
- 你在 Premise 3 那里 PARTIALLY DISAGREE 而不是直接 AGREE。本来你想说"用户不需要 Slack 生态"，但你又回忆起"5 个团队已经在用 webhook 把外部系统接进来"。这种"我本来想这么说，但数据告诉我其实是那样"的自我纠正，是 founder signal 里最强的"观察优于假设"信号。

- 你在 Open Questions 里写了"GFW 风险"，但没立刻去预防性建设国内 CDN fallback。这种"识别风险但不在证据出现前预防性投入"的克制，是 senior 创始人的标志，也是 user CLAUDE.md §3"只实现需求，不额外加功能"的体现。
